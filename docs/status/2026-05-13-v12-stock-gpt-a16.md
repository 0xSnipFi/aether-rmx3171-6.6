# Status snapshot - 2026-05-13 v12 stock-GPT A16 alignment

## What changed

- Default Android board target is now the real RMX3171 stock layout:
  boot-header-v2, physical `boot`, physical `dtbo`, non-A/B.
- `BoardConfig.mk` now includes `BoardConfigA16Legacy.mk` by default.
- `BoardConfigA16.mk` remains available only when
  `AETHER_BOOT_HEADER_VERSION=4` is explicitly set for PGPT-remap experiments.
- `fstab.mt6768.a16` no longer uses `slotselect`; RMX3171 stock boot has no
  slot suffix.
- `init.aether_root.rc` is now copied and imported so KernelSU/Magisk/NetHunter
  runtime hooks are not dropped from Android builds.
- MTK camera ISP3 experimental module build is default-on for release/module
  builds so physical testers can actually probe camera bring-up.
- Release notes now document recovery sideload and optional `dtbo.img` flash;
  the script no longer tells users to fastboot-flash an AnyKernel zip as boot.

## Current source readiness

This is a source-ready Android 16 / Linux 6.6 bring-up candidate for Narzo 30A,
not a hardware-proven daily driver yet.

Source-side pieces present:

- Linux 6.6 base with AETHER RMX3171 overlay.
- RMX3171 stock-GPT Android 16 board config.
- DTBO build path.
- Logical `vendor_dlkm` and `system_dlkm` config for custom super builds.
- First-stage and late module manifests.
- Panel/touch/audio/gauge/fingerprint/charging/FM slim AETHER modules.
- ECCCI/CCCI cellular kernel module packaging path.
- Experimental ISP3 camera module packaging path.
- KernelSU, Magisk compatibility hooks, and NetHunter kernel config support.

## Not proven yet

No physical RMX3171 boot log is committed. Therefore these remain unverified:

- display/backlight
- touch
- audio speaker route and SIA81xx PA
- charging current/thermal safety
- fingerprint HAL/TEE enrollment
- WiFi/BT/FM firmware bring-up
- cellular SIM/calls/data/VoLTE
- Mali G52 full Android GLES performance
- MTK ISP3 camera preview/photo

## Honest next gate

Build with Android clang-r510928 or newer, flash on an unlocked RMX3171, then
collect:

- boot dmesg
- `/proc/cmdline`
- `/sys/firmware/devicetree/base/model`
- `lsmod`
- `logcat -b all`
- display/touch/audio/charging/fingerprint/radio test notes

Only after that device loop should this tree be called production daily-driver.
